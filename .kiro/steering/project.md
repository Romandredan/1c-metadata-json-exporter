# Правила проекта 1c-metadata-json-exporter

## Что это за проект

Внешняя обработка 1С, которая выгружает структуру метаданных конфигурации в JSON для использования как контекста для LLM. Основной артефакт проекта — **стабильный формат JSON-выхода**, а не код обработки. Любое изменение формата — это продуктовое решение, а не рефакторинг.

## Инварианты

- **Исходный код обработки — единственное место логики выгрузки.** Не дублируем бизнес-правила в документации. Описание в `doc/format.md` и `schema/metadata-output.schema.json` синхронизируется с кодом, а не наоборот.
- **Формат JSON — публичный контракт.** Удаление полей, переименование ключей и смена семантики — **breaking change**. В будущем фиксируется в `CHANGELOG.md` как `BREAKING`.
- **Имена ключей JSON — на русском, в PascalCase** (как сейчас: `Конфигурация`, `Объекты`, `ПолноеИмя`, `СтандартныеРеквизиты`). Это согласуется с терминологией 1С и понятно LLM. Не меняем на транслит или английский без веской причины.
- **Рекурсия свёртывается в маркер** `{ "ПолноеИмя": "...", "Рекурсия": true }`. Глубина раскрытия ссылочных типов — 1..3, больше не даём.
- **Компактный JSON** (без отступов) — для передачи в LLM; читаемый JSON — для примеров и отладки. Оба формата должны быть эквивалентны по содержанию.

## Структура и именование

- Исходник обработки лежит в `src/epf/<ИмяОбработки>/`. Парный `<ИмяОбработки>.xml` — на том же уровне, рядом с каталогом.
- Новые внешние обработки — только в `src/epf/`. Не плодим новые верхнеуровневые папки без обсуждения.
- Пути в коде — через утилиту `ОбъединитьПуть`, не через конкатенацию со слэшами.
- Имена файлов выгрузки — через `БезопасноеИмяФайла`, чтобы исключить недопустимые символы и зависимость от регистра ФС.

## Стиль кода BSL

- Соблюдаем `.bsl-language-server.json`: длина строки ≤ 140, циклическая сложность ≤ 10, когнитивная сложность ≤ 15.
- Язык идентификаторов — **русский** (домен на русском, платформа на русском — смешивать не надо).
- Экспортные функции обработки обязаны иметь комментарий по стандарту **№453** (описание, параметры, возвращаемое значение). Подробности — см. steering `commenting-for-1c`.
- Служебные функции в областях `#Область СлужебныеПроцедурыИФункции_*` — без обязательного полного комментария, но с коротким пояснением назначения, если поведение неочевидно.
- `Попытка ... Исключение` без реакции (немое подавление) допустимо **только** там, где проверяем наличие свойства/коллекции у объекта метаданных (полиморфизм платформы). В остальных местах — либо логировать в `ЖурналРегистрации`, либо пробрасывать.

## Работа AI-агента в этом проекте

- Перед правкой кода обработки сверяться с актуальным `src/epf/ВыгрузкаСтруктурыМетаданныхВJSON/Ext/ObjectModule.bsl`. Это источник правды.
- Перед правкой формата выгрузки — сверяться с `schema/metadata-output.schema.json` и `doc/format.md`. Если правим код, меняющий формат — синхронно правим схему и документацию в том же коммите.
- Тесты автоматически не добавлять. Если требуется проверка — предложить пользователю, не создавать молча.
- Эталонные примеры JSON (`examples/`) — регрессионный материал. При изменении формата старые примеры перегенерировать и закоммитить вместе с изменением кода.
- Для любых срезов метаданных выбирать режим, который даёт **минимально достаточный** контекст: обычно «Подсистема» с глубиной 1 и `РежимДвиженийРегистров = "Коротко"`. Полное раскрытие — только под явный запрос.

## Совместимость

- Платформа: 1С:Предприятие **8.3.27+**.
- Работа с объектами расширений и регистрация в БСП («ДополнительныеОтчётыИОбработки») — штатный сценарий, не ломаем при рефакторинге.
- Кодировка: UTF-8 без BOM. Переводы строк в BSL/OS/XML — CRLF (см. `.gitattributes`).
