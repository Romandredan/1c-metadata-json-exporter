# Подумай, какую еще информацию для LLM стоит добавить в выгрузку? Что будет тебе полезно получить?


## Критически не хватает (даст прирост качества ответов в разы)

### 1. Свойства объектов метаданных — не только «что есть», но и «как устроено»

Для **справочника**: иерархический или нет, вид иерархии, владельцы, серии кодов, длина кода/наименования, использование групп, основная форма — по этому я понимаю, как правильно писать запросы и код.

```json
"СвойстваОбъекта": {
  "Иерархический": true,
  "ВидИерархии": "ИерархияГруппИЭлементов",
  "ДлинаКода": 11,
  "ТипКода": "Строка",
  "ДлинаНаименования": 150,
  "Владельцы": ["Справочник.Контрагенты"],
  "СерииКодов": "ВоВсемСправочнике"
}
```

Для **документа**: использование нумератора, периодичность нумерации, проведение, удаление движений, основание для ввода.

Для **регистра накопления**: вид регистра (Остатки/Обороты) — **критично**, без этого я могу написать запрос к `.ОстаткиИОбороты()` у оборотного регистра, что будет ошибкой.

Для **регистра сведений**: периодичность, режим записи (Независимый/ПодчинениеРегистратору), ведущие измерения.

Для **регистра бухгалтерии**: корреспонденция, связанный план счетов, признаки учёта.

**Это самое ценное добавление.** Без этих свойств я часто вынужден писать запросы «в общем виде», который может не подойти конкретному объекту.

### 2. Подчинение и иерархические связи

- У справочника — список владельцев (сейчас частично есть через стандартные реквизиты, но явный блок лучше).
- У документа — журналы, в которые он входит.
- У регистра — регистраторы (для регистров, подчинённых регистратору).
- У задачи — связанный бизнес-процесс.

```json
"Связи": {
  "Регистраторы": ["Документ.РеализацияТоваровУслуг", "Документ.ПоступлениеТоваровУслуг"],
  "Журналы": ["ЖурналДокументов.Продажи"]
}
```

Это даёт мне понимание **экосистемы вокруг объекта** — я сразу вижу, какие документы формируют движения в регистре, и могу предложить правильную цепочку обработки.

### 3. Формы — хотя бы список и назначение

Полные модули форм выгружать не надо (это объём), но **перечень форм с их назначением** — очень полезно:

```json
"Формы": [
  {"Имя": "ФормаСписка",    "Назначение": "ФормаСпискаСправочника"},
  {"Имя": "ФормаЭлемента",  "Назначение": "ФормаЭлементаСправочника"},
  {"Имя": "ФормаВыбора",    "Назначение": "ФормаВыбораСправочника"},
  {"Имя": "ФормаГруппы",    "Назначение": "ФормаГруппыСправочника"}
]
```

Полезно, когда пользователь просит «открыть форму такую-то» или «добавь кнопку на форму элемента» — я понимаю, какая форма есть, а какую надо создавать.

### 4. Макеты и команды

```json
"Макеты": [{"Имя": "ПечатьСчета", "ТипМакета": "ТабличныйДокумент"}],
"Команды": [{"Имя": "ОткрытьОтчёт", "Группа": "НавигационнаяПанельВажное"}]
```

Это помогает при запросах про печать, отчёты, интерфейсные расширения.

### 5. Роли и права на объект (сводно)

Не полная выгрузка всех ролей (это отдельная задача), а **для каждого объекта** — список ролей, у которых есть право на этот объект, и какое именно:

```json
"ПраваНаОбъект": [
  {"Роль": "ПолныеПрава",      "Права": ["Чтение","Изменение","Добавление","Удаление"]},
  {"Роль": "МенеджерПродаж",   "Права": ["Чтение","Добавление"], "ОграничениеRLS": true}
]
```

При запросах про RLS и разграничение доступа — это золото.

## Сильно помогает (средний приоритет)

### 6. Модули — карта API без тел процедур

Выгружать **сигнатуры** экспортных процедур/функций (имя, параметры, тип возврата, строка-описание из комментария над ней) — для модулей менеджера, модулей объекта, общих модулей, модулей набора записей.

```json
"Модули": {
  "МодульМенеджера": {
    "ЭкспортныеПроцедуры": [
      {
        "Имя": "СформироватьПечатнуюФорму",
        "Параметры": "МассивОбъектов, ИмяМакета",
        "ВозвращаемыйТип": "ТабличныйДокумент",
        "Описание": "Формирует печатную форму по шаблону."
      }
    ]
  }
}
```

Даёт мне **API-карту конфигурации** — я сразу вижу, что можно переиспользовать, а не писать с нуля. **Без тел процедур** — никакого риска с объёмом.

### 7. Подписки на события, относящиеся к объекту

Сейчас полная картина реактивной логики остаётся для меня чёрным ящиком. Если в выгрузке указать:

```json
"ПодпискиНаСобытия": [
  {"Имя": "ПроверкаНоменклатурыПриЗаписи", "Событие": "ПередЗаписью", "Обработчик": "ОбщийМодуль.ПроверкаДанных.Проверить"}
]
```

— я пойму, что при записи объекта срабатывает такая-то логика, и учту это в рекомендациях.

### 8. План обмена — узлы и состав

Для планов обмена — список узлов (хотя бы количество и имена автоматически отмечаемых реквизитов), и какие объекты в них входят (это можно и отдельной секцией на уровне корня).

### 9. Критерии отбора

```json
"КритерииОтбора": [{"Имя": "ДокументыКонтрагента", "Состав": ["Документ.Реализация", "Документ.Оплата"]}]
```

Помогает с запросами вида «покажи все документы по контрагенту».

### 10. Регламентные задания, связанные с объектом

Если в конфигурации есть РЗ, которые массово обрабатывают этот объект, — упомянуть их имя и расписание (без подробного расписания, хватит периодичности).

## Полезно, но объёмно (низкий приоритет — по запросу)

### 11. Выборка реальных данных — примеры значений

Для понимания **как заполнен** справочник/регистр — 3–5 примеров записей (с анонимизацией чувствительных полей). Это **радикально** меняет качество ответов по данным, потому что я перестаю гадать «а что там за формат кода? а какие значения в перечислении-подобном реквизите на самом деле используются?».

Делать **опциональным** параметром (`ВключатьПримерыДанных = Ложь` по умолчанию) — чтобы не раздувать вывод и не создавать утечек чувствительных данных.

### 12. Статистика использования

Количество записей в таблице, дата последней записи, самая ранняя запись — даёт мне масштаб. «Справочник на 50 записей» и «справочник на 3 миллиона записей» — это два разных мира, и запросы к ним пишутся по-разному.

```json
"Статистика": {
  "КоличествоЗаписей": 47832,
  "ДиапазонДат": {"Мин": "2019-01-05", "Макс": "2026-05-06"}
}
```

Опционально, может быть медленно на больших таблицах — нужен флаг.

### 13. Индексы

Список реквизитов, по которым объявлен индекс (свойство «Индексирование» у реквизита + индексы формы регистра). Помогает мне **не предлагать неоптимальные запросы** и видеть, где нужна оптимизация.

### 14. Параметры сеанса и константы, связанные с функциональностью

Контекст — для каких настроек конфигурации предназначен объект.

## Метауровень — свойства конфигурации целиком

В корне JSON-файла помимо `Конфигурация` и `Версия` полезно видеть:

```json
"Метаданные": {
  "ОсновнойЯзык": "Русский",
  "РежимСовместимости": "Версия8_3_21",
  "РежимУправляемогоПриложения": true,
  "РежимИспользованияМодальности": "НеИспользовать",
  "ВариантВстроенногоЯзыка": "Русский",
  "ХранилищеВариантовОтчетов": "...",
  "ПодключённыеРасширения": [
    {"Имя": "МоёРасширение", "Версия": "1.0", "Активно": true}
  ]
}
```

**Зачем**: я пойму версию платформы и могу/не могу предлагать новые синтаксические конструкции, использовать асинхронные методы, модальные окна и т. п. Без этого я рискую предложить код, который не скомпилируется в вашей базе.

## Мой рейтинг — с чего начать

Если делать по нарастающей, я бы пошёл так:


| Приоритет | Блок | Объём кода | Прирост качества |
| :-- | :-- | :-- | :-- |
| 1 | Свойства объектов (иерархичность, вид регистра, периодичность…) | ~50 строк | Очень высокий |
| 2 | Метаданные конфигурации (режим совместимости, расширения) | ~20 строк | Высокий |
| 3 | Связи (регистраторы, владельцы, журналы) | ~40 строк | Высокий |
| 4 | Формы, макеты, команды (перечень) | ~30 строк | Средний |
| 5 | Роли и права на объект (сводно) | ~50 строк | Высокий для задач безопасности |
| 6 | Модули — карта экспортных процедур | ~80 строк | Очень высокий |
| 7 | Подписки на события объекта | ~30 строк | Средний |
| 8 | План обмена, критерии отбора, РЗ | ~40 строк | Средний |
| 9 | Примеры данных (опционально) | ~40 строк | Очень высокий, но риск утечки |
| 10 | Статистика и индексы | ~30 строк | Средний |

## Что я бы сделал в первую очередь

**Блоки 1, 2, 3** — самая высокая отдача при минимальном объёме работы. Это ~110 строк кода и радикальное улучшение качества моих ответов на типовые задачи разработки.

**Потом блок 6** — карта API. Это превратит вашу обработку из «описания структуры» в полноценный **контекстный файл для LLM-ассистента разработчика**.

## Принципиальный совет по формату

Для всех новых блоков — **делайте включение опциональным** через флаги в `НовыеПараметрыВыгрузки`:

```bsl
Параметры.Вставить("ВключатьСвойстваОбъектов", Истина);   // по умолчанию — да, дешево
Параметры.Вставить("ВключатьСвязи",           Истина);   // по умолчанию — да
Параметры.Вставить("ВключатьФормыМакетыКоманды", Истина);// по умолчанию — да
Параметры.Вставить("ВключатьПраваНаОбъект",   Ложь);     // по умолчанию — нет, объёмно
Параметры.Вставить("ВключатьКартуАPI",        Ложь);     // по умолчанию — нет, медленно
Параметры.Вставить("ВключатьПодпискиНаСобытия", Ложь);
Параметры.Вставить("ВключатьПримерыДанных",   Ложь);     // по умолчанию — нет, чувствительные данные
Параметры.Вставить("ВключатьСтатистику",      Ложь);     // по умолчанию — нет, медленно
```

Это позволит мне выбирать режим: «компактный контекст для быстрого запроса» vs «полный контекст для глубокой задачи рефакторинга».

## Предложение

Давайте начнём с **блоков 1–3**. Я готов спроектировать:

1. Расширение структуры `Описание` новыми полями.
2. Функции-наполнители для каждого вида метаданных (отдельно для справочника, отдельно для документа и т. д.) — чтобы код остался читаемым.
3. Обновление параметров выгрузки и обратную совместимость (чтобы существующие вызовы API продолжали работать без изменений).

Объём — одна итерация, 30–40 минут работы. Если согласны — подтвердите, и я пришлю конкретный код.

И да, если планируете публиковать обработку для сообщества — именно эти блоки сделают её **уникальным инструментом**. На Инфостарте похожих обработок немало, но большинство из них ограничиваются базовым скелетом. Инструмент, специально заточенный под подготовку контекста для LLM, — это ниша, которая только формируется, и занять её правильными фичами сейчас очень выгодно.
