# Формат JSON-выгрузки

Контракт описывает структуру файлов, которые создаёт обработка `ВыгрузкаСтруктурыМетаданныхВJSON`. Машинно-проверяемая версия — `schema/metadata-output.schema.json`. Источник правды — `src/epf/ВыгрузкаСтруктурыМетаданныхВJSON/Ext/ObjectModule.bsl`. Этот документ обязан меняться синхронно с кодом и схемой.

## Общие правила

- Все ключи — **на русском, в PascalCase**. Это совпадает с терминологией платформы 1С и делает выгрузку самодокументируемой для LLM.
- **Рекурсия всегда свёртывается** в маркер `{ "ПолноеИмя": "...", "Рекурсия": true }`. Это позволяет агенту отличить «обрыв по глубине» от «уже видели этот тип».
- Глубина раскрытия ссылочных типов — **1..3**. Жёсткий верх нужен, чтобы выгрузка не взрывалась для связанных сущностей.
- Кодировка файлов — **UTF-8**. Даты пишутся платформой через `ФорматДатыJSON.ISO` — это ISO-8601 **без таймзоны**, пример: `"2026-05-11T00:07:07"`.
- Есть два эквивалентных представления: компактное (без отступов, для отправки в LLM) и с отступами `Символы.Таб` (для отладки/git-дифов). По содержанию они идентичны.

## Два варианта верхнего уровня

Верхний уровень — `oneOf`:

### 1. Общий файл (`ОтдельныеФайлы = Ложь`)

```jsonc
{
  "Конфигурация": "УправлениеТорговлей",
  "Версия":       "11.5.22.129",
  "ДатаВыгрузки": "2026-05-11T00:07:07",
  "Подсистема":   { /* опционально, только при Режим = "Подсистема" */ },
  "Объекты":      [ /* массив описаний объектов */ ]
}
```

### 2. Отдельный файл на объект (`ОтдельныеФайлы = Истина`)

```jsonc
{
  "Подсистема": { /* опционально, если файл создан в рамках режима "Подсистема" */ },
  "Объект":     { /* одно описание объекта */ }
}
```

Реальные примеры обеих форм лежат в `examples/`:

- `Документ.ЗаказКлиента_20260511_000707.json` — общий файл, режим «Один» для документа со всеми ТЧ и движениями по 10 регистрам.
- `Подсистема.Доработки_КомандныйИнтерфейс.Подсистема.Администрирование.Подсистема.Константы_20260511_000616.json` — общий файл, режим «Подсистема», с блоком `Подсистема` на верхнем уровне и объектами вида `Константа`/`РегистрСведений`/`РегистрНакопления`.

Отличие двух режимов — не только в структуре файла, но и в **политике защиты от рекурсии**:

- В общем файле `ПосещенныеТипы` — один на весь файл. Повторная встреча типа даёт маркер рекурсии.
- В режиме отдельных файлов `ПосещенныеТипы` **сбрасывается для каждого объекта** — чтобы каждый файл оставался самодостаточным контекстом для LLM.

## Блок «Подсистема»

```json
{
  "Имя":           "Константы",
  "Синоним":       "Значения по умолчанию",
  "ПолноеИмя":     "Подсистема.Доработки_КомандныйИнтерфейс.Подсистема.Администрирование.Подсистема.Константы",
  "ПутьРодителей": ["Доработки_КомандныйИнтерфейс", "Администрирование"]
}
```

`ПутьРодителей` упорядочен от корня к текущей подсистеме (саму подсистему не включает). Для корневой подсистемы — пустой массив.

## Блок «Объект»

Универсальное описание любого вида метаданных. Набор ключей зависит от вида.

Всегда присутствуют:

| Поле | Тип | Смысл |
|---|---|---|
| `ВидОбъекта` | строка | Префикс полного имени: `Справочник`, `Документ`, `Перечисление`, `Константа`, `РегистрСведений`, `РегистрНакопления`, `РегистрБухгалтерии`, `РегистрРасчета`, `ПланВидовХарактеристик`, `ПланСчетов`, `ПланВидовРасчета`, `Подсистема` и т. д. |
| `Имя` | строка | `ОбъектМД.Имя` |
| `Синоним` | строка | `ОбъектМД.Синоним` (может быть пустым) |
| `ПолноеИмя` | строка | Например `Документ.РеализацияТоваровУслуг` |
| `Комментарий` | строка | `ОбъектМД.Комментарий` (может быть пустым) |
| `СтандартныеРеквизиты` | массив | `Ссылка`, `Дата`, `Номер`, `Проведен`, `ПометкаУдаления` и т. п. Для объектов без стандартных реквизитов — пустой массив |

Опционально, в зависимости от вида:

| Поле | Когда присутствует |
|---|---|
| `Реквизиты` | У видов с коллекцией `Реквизиты` (Справочники, Документы, Регистры, ПВХ, ПВР, ПС). |
| `ТабличныеЧасти` | У видов с коллекцией `ТабличныеЧасти` (Справочники, Документы, …). |
| `Движения` | Только для Документов. |
| `Измерения` | Только для регистров всех четырёх видов. |
| `Ресурсы` | Только для регистров с коллекцией `Ресурсы`. |
| `Предопределённые` | При `ВключатьПредопределенные = Истина`, для Справочников / Перечислений / ПВХ / ПВР / ПС. Массив имён. |

## Реквизит (реквизит, измерение, ресурс)

```jsonc
{
  "Имя":          "Контрагент",
  "Синоним":      "Контрагент",
  "Комментарий":  "",
  "Типы":         ["Справочник.Контрагенты"],
  "ПроверкаЗаполнения": "Выдавать ошибку",

  // Квалификаторы типа — присутствуют только если платформа отдала значение:
  "Длина":       10,     // КвалификаторыЧисла.Разрядность
  "Точность":    2,      // КвалификаторыЧисла.РазрядностьДробнойЧасти
  "ДлинаСтроки": 150,    // КвалификаторыСтроки.Длина

  // Только при РежимСсылочныхТипов = "Раскрывать" и Уровень < ГлубинаВложенности:
  "Раскрытие": [
    { /* вложенный Объект */ },
    { "ПолноеИмя": "Справочник.Контрагенты", "Рекурсия": true }
  ]
}
```

Поле `Типы` — массив строк. Каждый элемент — либо **полное имя ссылочного объекта** (`Справочник.Контрагенты`), либо **имя платформенного/примитивного типа** (`Строка`, `Число`, `Дата`, `Булево`, `УникальныйИдентификатор`).

## Табличная часть

```jsonc
{
  "Имя":       "Товары",
  "Синоним":   "Товары",
  "Реквизиты": [ /* массив Реквизитов */ ]
}
```

## Движение (регистр-приёмник документа)

```jsonc
{
  "Вид":       "РегистрНакопления",
  "Имя":       "ТоварыНаСкладах",
  "Синоним":   "Товары на складах",
  "ПолноеИмя": "РегистрНакопления.ТоварыНаСкладах",

  // Только при РежимДвиженийРегистров = "Подробно":
  "Измерения": [ /* Реквизиты */ ],
  "Ресурсы":   [ /* Реквизиты */ ],
  "Реквизиты": [ /* Реквизиты */ ]
}
```

При `РежимДвиженийРегистров = "Коротко"` остаются только `Вид`, `Имя`, `Синоним`, `ПолноеИмя` — этого достаточно, чтобы LLM понимал, какие регистры движутся документом, и при необходимости запросил подробности отдельной выгрузкой.

## Маркер рекурсии

```json
{ "ПолноеИмя": "Справочник.Контрагенты", "Рекурсия": true }
```

Встречается только внутри массива `Раскрытие`. `Рекурсия` — всегда `true` (в схеме закреплено через `const`).

## Правила именования файлов

- Режим **Один**: `<БезопасноеИмяПолногоИмени>.json`, например `Документ.РеализацияТоваровУслуг.json`.
- Режим **Список**: `metadata_list_<yyyyMMdd_HHmmss>.json` — метка времени уже встроена в имя и отдельно не добавляется.
- Режим **Подсистема**: `<БезопасноеИмяПолногоИмени>.json`, при `ДобавлятьВременнуюМетку = Истина` — с постфиксом `_<yyyyMMdd_HHmmss>`.
- Запрещённые на Windows символы (`" * / \ : < > ? |`) заменяются на `_` функцией `БезопасноеИмяФайла`.

## Совместимость и эволюция формата

- Добавление новых **опциональных** ключей — не ломающее изменение.
- Переименование ключей, смена типа значения, удаление поля — **breaking**. Фиксировать в `CHANGELOG.md` с префиксом `BREAKING:` и перегенерировать примеры в `examples/`.
- В схеме стоит `additionalProperties: false` для всех объектов — это позволяет ловить случайные «утечки» новых полей и выносить их в формат осознанно.


## Валидация примеров по схеме

Файлы в `examples/` считаются **эталонными** и должны проходить валидацию по `schema/metadata-output.schema.json`. Если после правок обработки какой-то пример перестаёт валидироваться — это либо надо считать breaking-изменением формата (и тогда править схему + `CHANGELOG.md`), либо откат в коде.

Разовая проверка без добавления зависимостей в проект — через `ajv-cli` в изолированной папке:

```powershell
# один раз
New-Item -ItemType Directory -Path _tmp_pkg -Force | Out-Null
'{ "type": "module" }' | Set-Content _tmp_pkg/package.json -Encoding UTF8
npm --prefix _tmp_pkg install ajv ajv-formats --silent

# прогон
node -e "import('ajv/dist/2020.js').then(async m=>{const Ajv=m.default;const af=(await import('ajv-formats')).default;const fs=await import('node:fs');const path=await import('node:path');const ajv=new Ajv({allErrors:true,strict:false});af(ajv);const v=ajv.compile(JSON.parse(fs.readFileSync('schema/metadata-output.schema.json','utf8')));let ok=true;for(const f of fs.readdirSync('examples').filter(x=>x.endsWith('.json'))){const d=JSON.parse(fs.readFileSync(path.join('examples',f),'utf8'));if(v(d))console.log('OK  '+f);else{ok=false;console.log('ERR '+f);for(const e of v.errors.slice(0,10))console.log('    '+e.instancePath+' '+e.message);}}process.exit(ok?0:1);})" --experimental-vm-modules
```

Проверка намеренно не оформлена как пакет-зависимость проекта: `1c-metadata-json-exporter` — это 1С-проект, а не Node-пакет. Если потребуется автоматизация в CI, вынесите валидацию в `tools/check-examples.ps1`.
