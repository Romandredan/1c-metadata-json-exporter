# 1c-metadata-json-exporter

Внешняя обработка для 1С, которая выгружает **структуру метаданных конфигурации в JSON**. Назначение — дать LLM-агенту (Kiro, Copilot, ChatGPT и т. д.) компактный и предсказуемый контекст о прикладной модели конфигурации, без необходимости открывать конфигуратор или загружать исходники.

Основной артефакт проекта — не сама обработка, а **формат её выхода**. Всё остальное (структура репозитория, тесты, документация) подчинено стабильности этого формата.

## Быстрый старт

1. Открыть файл обработки в 1С:Предприятии (режим тонкого/толстого клиента):
   `src/epf/ВыгрузкаСтруктурыМетаданныхВJSON/` — это исходник в формате Конфигуратора. Для использования в запущенной ИБ соберите `.epf` через Конфигуратор или EDT.
2. В открытой обработке выбрать режим: **один объект**, **список объектов** или **подсистема**.
3. Указать папку выгрузки и запустить.
4. На выходе получить один JSON-файл (общий) или набор файлов (по одному на объект).

Программное использование из кода другой обработки:

```bsl
Обработка = ВнешниеОбработки.Создать("C:\Путь\ВыгрузкаСтруктурыМетаданныхВJSON.epf");

// Вариант 1: в файл.
Результат = Обработка.ВыгрузитьОдинОбъект(
    "Документ.РеализацияТоваровУслуг",
    "C:\Temp\");

// Вариант 2: получить структуру без записи в файл.
Описание = Обработка.ПолучитьОписаниеОбъекта("Документ.РеализацияТоваровУслуг");
```

## Что умеет обработка

- Три режима выбора состава: **Один** объект, **Список** объектов, **Подсистема** (с опцией «включая вложенные подсистемы»).
- Универсальное описание любого вида метаданных: справочники, документы, перечисления, ПВХ/ПВР/ПС, регистры сведений/накопления/бухгалтерии/расчёта, подсистемы.
- Раскрытие ссылочных типов с регулируемой глубиной (1–3) и защитой от рекурсии через маркер `{ ПолноеИмя, Рекурсия: true }`.
- Детализация движений документов: **Коротко** (только ссылки на регистры) или **Подробно** (с измерениями, ресурсами и реквизитами регистров).
- Включение/исключение имён предопределённых элементов.
- Опция «Компактный JSON» — без отступов, для прямой отправки в LLM.
- Две стратегии файлов: **общий файл** (единый контекст для защиты от рекурсии) или **отдельный файл на объект** (каждый файл максимально полный).
- Регистрация в подсистеме БСП «ДополнительныеОтчётыИОбработки» (метод `СведенияОВнешнейОбработке()`).

## Формат выгрузки

Верхний уровень — объект с ключами:

- `Конфигурация` — имя конфигурации.
- `Версия` — версия конфигурации.
- `ДатаВыгрузки` — дата и время снятия дампа.
- `Подсистема` — блок описания подсистемы (только для режима «Подсистема»).
- `Объекты` — массив описаний объектов (общий файл) **или**
- `Объект` — одно описание объекта (отдельный файл).

Подробное описание каждого поля, включая квалификаторы типов и признаки обязательности, живёт в `doc/format.md`, а машинно-проверяемый контракт — в `schema/metadata-output.schema.json`. Оба файла — источник правды для AI-агентов и регрессий.

## Разработка

- Рабочий формат хранения обработки — **исходники Конфигуратора** (`Ext/ObjectModule.bsl` + `Forms/`). Собранный `.epf` в git не коммитим (см. `.gitignore`).
- Стиль кода и диагностики — `.bsl-language-server.json` (язык `ru`, длина строки 140, без обязательных описаний в общих модулях, проверка вложенности в тело модуля).
- Комментарии к процедурам и функциям — по стандарту **№453** (см. steering `commenting-for-1c`).
- Менять формат JSON — только осознанно и с пометкой **breaking** в будущем `CHANGELOG.md`. Сам стабильный формат — важнее кода обработки.

## Совместимость

- Платформа 1С:Предприятие **8.3.27+**.
- Работает в любой конфигурации; отдельно обрабатывает объекты расширений и регистрацию в БСП.
- Кодировка BSL-файлов — UTF-8, переводы строк CRLF.
